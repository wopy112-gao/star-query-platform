# 星宝数据平台 —— 全量数据修复执行方案

## 问题

1. 早期合并脚本 `drop_duplicates(subset=['场景ID'])` 导致多疾病展开丢失
2. 本地场景数 2,498,996 vs ClickHouse 源 2,511,123，差 12,127 个场景
3. 彩蛋映射 30,747 条只 merge 上 26,907 条

## 方案：按月分片全量重拉 + 彩蛋合并

### 分片计划

共 11 片，覆盖 2025-08 ~ 2026-07：

| 分片 | 展开行数 | 预计耗时 |
|:----:|:--------:|:--------:|
| 2025-08 | 525 | ~1s |

| 2025-11 | 13,506 | ~1s |
| 2025-12 | 31,404 | ~1s |
| 2026-01 | 53,112 | ~1s |
| 2026-02 | 51,346 | ~1s |
| 2026-03 | 46,685 | ~1s |
| 2026-04 | 356,220 | ~3s |
| 2026-05 | 923,035 | ~8s |
| 2026-06 | 922,195 | ~8s |
| 2026-07 | 402,932 | ~4s |
| **合计** | **2,804,250** | **~30s** |

### 执行步骤

```
┌─────────────────────────────────────────┐
│ Step 1 │ 备份当前全量 + 彩蛋映射          │
│ Step 2 │ 按月循环：拉取 → 合并 → 验证     │
│ Step 3 │ 拉取彩蛋映射 → merge 到全量      │
│ Step 4 │ 重建 DuckDB + 重启               │
│ Step 5 │ 验证：行数/多疾病/彩蛋/查询      │
└─────────────────────────────────────────┘
```

### 防超时机制

- 每片设置 `send_receive_timeout=60s`
- 某片失败时自动重试（最多 3 次）
- 某片持续失败时跳过，不影响已完成的月份
- 所有片完成后统一替换全量文件

### 回滚方案

- 备份文件：`/root/All_data_ch_full.parquet.bak` + `data/egg_mapping.parquet.bak`
- 重建回滚只需 1 分钟：恢复备份 → 重建 DuckDB → 重启

### 验证清单

- [ ] 总行数 ≈ 2,804,250
- [ ] 场景数 ≈ 2,511,123
- [ ] 多疾病场景 ≈ 205,513
- [ ] 疾病名称唯一值 ≈ 322（或接近增量中的值）
- [ ] 彩蛋覆盖 >= 30,000
- [ ] 查询功能：按疾病筛选、按疾病大类筛选正常
- [ ] 下载功能：筛选导出正常
