# 2026-05-25 变更说明

## 一、方向三讨论：Prompt 拆解（完整版）

**决策：** 选择完整版（方向 B），全链路结构化 Pipeline

### 全链路架构

```
用户问题 → ①意图拆解器 → ②结构化缓存 → ③模板匹配(四元组) → ④LLM精简(兜底) → ⑤SQL质量门禁 → 执行
```

### 讨论确认的决策点

| 序号 | 决策内容 | 结论 |
|------|---------|------|
| 1 | 兜底路径 | 必须用 LLM，两级兜底（精简翻译 → 展示意图给用户确认） |
| 2 | 意图拆解调用方式 | 独立 DeepSeek 调用（后端自包含） |
| 3 | 多实体/多条件模型 | 扩展 conditions 列表，支持 disease/drug_any/named_not_ordered/mentioned_not_named/mn_no_named_ordered 等 15+ 条件类型 |
| 4 | query_type 扩展 | 拆为 query_pattern（8种）+ agg（17种）双维枚举 |
| 5 | 模板匹配粒度 | 细粒度优先 + 通配符兜底 |
| 6 | 模板盘点 | 输出完整模板清单（~50个模板），按 pattern 分类 |
| 7 | SQL 质量门禁 | 语法错误 → 拦截重试；维度/口径错配 → 拦截修正；结果0行/过多 → 标注不拦截 |
| 8 | 缓存改造 | 新 key = md5(结构化JSON)，旧缓存清空 |
| 9 | 前端展示 | 查询框下方展示意图拆解结果 |
| 10 | 模板未命中的两级兜底 | 第一级：精简 Prompt LLM翻译(~500 tokens)；第二级：展示意图让用户确认后重试 |

## 二、测试环境 bug 修复

### 修复 1：`Optional` 未导入导致进程启动失败

- **文件：** `star-query-test/backend/query_router.py`
- **问题：** 第 446 行 `def _query_fallback(question: str, scope_hint: Optional[str], ...)` 使用了 `Optional` 但文件顶部未从 `typing` 导入
- **修复：** 文件顶部添加 `from typing import Optional`
- **影响：** 测试环境首次安全重启失败，补上后成功

### 修复 2：意图拆解器将"疾病"等泛指词当成实体值

- **文件：** `star-query-test/backend/query_intent.py`
- **问题：** 用户输入"疾病成交月度趋势"→ LLM 把"疾病"当成具体疾病名 → intent 生成 `{conditions: [{type: "disease", value: "疾病"}]}` → SQL 出现 `WHERE 疾病名称 LIKE '%疾病%'`
- **修复（两层）：**
  - **Prompt 层：** 增加规则说明，要求忽略"疾病""药品""场景""门店"等泛指词
  - **代码层：** 新增 `_filter_generic_conditions()` 后处理函数，过滤 `GENERIC_KEYWORDS = {"疾病", "药品", "药品类型", "场景", "门店", "商品", "产品", "品类"}`
- **测试：** "疾病成交月度趋势"修复后成功（泛指词被过滤，SQL 只保留 `WHERE 交易是否达成='是'`）

### 修复 3：SQL 渲染器产生双 WHERE

- **文件：** `star-query-test/backend/template_matcher.py` + `sql_renderer.py`
- **问题：** 部分模板（tx03/tx04/tx08/tx09/tt02）的 SQL 骨架有硬编码 `WHERE 交易是否达成='是'`，而 `{conditions}` 占位符被渲染为 `WHERE 疾病名称 LIKE '%感冒%'` → 两个 WHERE 冲突 → SQL 解析报错
- **修复（两个层面）：**
  - **渲染器：** 新增 `{conditions_extra}` 占位符，渲染为 `AND xxx`（不带 WHERE 前缀），与 `{conditions}`（渲染为 `WHERE xxx`）区分
  - **模板：** 5 个模板的 `{conditions}` → `{conditions_extra}`
- **受影响的模板：** tx03（疾病成交分布）、tx04（药品成交分布）、tx08（药品成交-城市分布）、tx09（药品成交-省份分布）、tt02（疾病成交月度趋势）
- **测试：** "感冒各城市成交场景数"修复后 SQL 正确：`FROM data WHERE 交易是否达成='是' AND 疾病名称 LIKE '%感冒%' ...`

## 六、INTENT_SYSTEM_PROMPT 缺表名修复（2026-05-25 14:35）

**问题：** 用户在星宝数据平台提问"心脑血管疾病中有阿司匹林的场景占比"，LLM 生成 `FROM 零售数据表`，被 SQL 引擎拦截（仅允许 `data`）

**根因：** v3 精简 Prompt `INTENT_SYSTEM_PROMPT`（兜底路径用）缺了表名声明，LLM 脑补了"零售数据表"。旧全量 Prompt 正确包含 `表名: {table_name}`。

**修复（2个文件）：**
| 文件 | 修改 |
|------|------|
| `star-query/backend/llm_translator.py` | `INTENT_SYSTEM_PROMPT` 新增 `数据表在 DuckDB 中注册为 \`data\`，SQL 中的 FROM 子句必须写 \`FROM data\`` |
| `star-query/backend/models.py` | `QueryInfo.source` Literal 类型加 `"llm_intent"`（之前只允许 template/moss/cache） |

**流程：** 测试环境先改 → `safe-restart.sh --test` 验证通过 → 同步到正式环境 → `safe-restart.sh` 重启生产
